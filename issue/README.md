# issue/

このディレクトリは、リポジトリ内で 1 件 1 ファイルの自己完結した issue を Markdown で管理する。
別セッションの誰か（人間 / コーディングエージェント）が**単独で読んで着手できる**ことをゴールとする。

> NOTE: 本リポジトリは「ローカル完結 issue 管理 CLI」そのものを開発するプロジェクトであり、
> この `issue/` 運用は将来 `issue` コマンドが自動化する対象でもある（ドッグフーディング）。

## 命名

- ファイル名: `N-slug.md`（例: `1-issue-cli-mvp.md`）
- `N` は **1 始まりの連番整数。ゼロ埋めしない**（`1`, `2`, …, `12345`）
- 一覧の並び順は `id` を**数値**としてソートして担保する（ファイル名の辞書順には依存しない。
  固定幅ゼロ埋めの桁あふれ問題を避けるため）
- 一度付けた番号・ファイル名はリネーム / 移動しない
- `slug` は title を英小文字ケバブケースで要約

## frontmatter

| key | 説明 |
| --- | --- |
| `id` | 1 始まりの連番整数。ゼロ埋めしない。ファイル名の `N` と一致 |
| `title` | 1 行サマリ |
| `status` | `open` / `closed`（GitHub 準拠の 2 値。ADR 0003） |
| `priority` | `P0` / `P1` / `P2` / `P3`（任意フィールド） |
| `created` / `updated` | `YYYY-MM-DD` |
| `labels` | 分類・関連領域の配列。種別（旧 `type`: `bug` / `feature` / `epic` …）も label で表現 |

> `type` / `related` フィールドは廃止（ADR 0003）。種別は `labels` に寄せ、
> 関連 issue は本文末尾の `## Related` セクションに `- #N` リンクで書く。

## 本文セクション

背景 / なぜ・現状(As-is)・あるべき姿(To-be)・受け入れ条件（チェックボックス）・
スコープ外・関連・決定事項・未解決の論点。エピックには「サブタスク」を加える。

**機能的な振る舞いで書く。** `file:line`・スキーマ・関数名などの実装詳細は陳腐化するため本文に持ち込まない。
「ユーザー / システムが何をするか・何が観測できるか」を書く。

## バッチ実行

open な issue をまとめて着手する場合は、通常セッションで `issue/` を読み、
priority 順に受け入れ条件を満たす実装を進める。
